# 完成物回传飞书方案分析

> 目标：NotebookLM 生成的图/PPT/PDF/视频、小红书发布链接、文档完成等，都要有「原始完成物」的回复，而不是只给总结或概述。

---

## 一、现状梳理

### 1.1 NotebookLM 可生成的 artifact 类型

| 类型 | 命令 | 输出格式 | 保存位置 |
|------|------|----------|----------|
| Slide Deck | `artifact_manager.py generate-slides --wait --output xxx.pdf` | PDF | `--output` 指定路径 |
| Infographic | `artifact_manager.py generate-infographic --wait --output xxx.png` | PNG | `--output` 指定路径 |
| Audio (podcast/briefing/debate) | `artifact_manager.py generate --format DEEP_DIVE --wait --output xxx.mp3` | MP3 | `--output` 指定路径 |
| Video | NotebookLM API 支持 `download_video` | 视频 | 同上 |

所有 artifact 在 `--wait` 完成后都会：
- 打印 `✅ Saved to: {path}` 到 stdout
- 或打印 `🔗 URL: {status['url']}`（NotebookLM 云端 URL）

### 1.2 小红书发布

- `XHSPublishResult` 有 `final_url` 字段
- 发布成功时 `result.final_url` 为笔记链接（如 `https://creator.xiaohongshu.com/...`）
- 通过 mcporter/exec 调用后，stdout 可能包含该链接，取决于 xhs-toolkit 的日志输出

### 1.3 当前 OpenClaw 的「完成物」传递机制

**MEDIA: 指令（已有）：**
- Agent 在最终回复中输出 `MEDIA:/path` 或 `MEDIA:https://...` 时，`parseReplyDirectives` 会解析并提取
- `loadWebMedia` 支持 `file://`、`/`、`./`、`https://` 等路径
- 各 channel 的 `sendMedia` 会把 media 发到目标（飞书通过 feishu-openclaw 插件的 `sendMediaMessage`）

**Subagent announce 流程（缺口）：**
1. Researcher 完成 → `roundOneReply` = 模型最后一条 assistant 消息
2. `runSubagentAnnounceFlow` 把 reply 写入临时 .md，并给主 agent 发 trigger message
3. Trigger 里只有 `MEDIA:${fullFindingsPath}`（.md 文件），**不包含** researcher 生成的 PDF/PNG/MP3 路径
4. 主 agent 收到 Findings 文本，需要**自己**在回复里加 `MEDIA:/path/to/slides.pdf` 才会发文件
5. 主 agent 未必会加，且 reply 里可能根本没有路径（模型可能只写「已生成」）

**Internal comm [回报]：**
- `maybeSendInternalCommAnnounce` 只发文本，不支持 `mediaUrl`/`mediaUrls`
- 即使 researcher 生成了文件，[回报] 也不会带附件

---

## 二、根因归纳

| 环节 | 问题 |
|------|------|
| **nblm SKILL** | 没有要求 researcher 在生成 artifact 后，在回复末尾输出 `MEDIA:<output_path>` |
| **exec 工具** | 工具结果里有 stdout，但模型不一定把路径提取并写入自己的回复 |
| **subagent announce** | 只从 reply 文本创建 .md，不解析 reply 中的 `MEDIA:` 行，也不把 exec 结果里的路径注入 |
| **internal comm** | `sendToInternalGroup` 支持 `mediaUrl`/`mediaUrls`，但 `maybeSendInternalCommAnnounce` 未传入 |
| **主 agent prompt** | 没有明确要求：若 Findings 中有本地文件路径或链接，必须在回复中附带 `MEDIA:` 或链接 |

---

## 三、Clawbot 现有能力（可复用）

1. **MEDIA: 解析**：`splitMediaFromOutput` 支持多行、多文件、`file://`、相对路径
2. **飞书发文件**：`feishu-openclaw` 的 `sendMediaMessage(client, chatId, filePath, caption)` 支持 `file://` 路径
3. **deliverOutboundPayloads**：`payload.mediaUrls` 会逐个调用 `sendMedia`，飞书走对应 adapter
4. **exec 工具**：返回 `{ stdout, stderr, exitCode }`，stdout 中有 `Saved to: /path` 等

## 四、需改动项（方案级，不涉及具体代码）

### 4.1 nblm SKILL 增强

**位置**：`~/.openclaw/skills/nblm/SKILL.md`、`nblm-1.0.1/.agent/skills/nblm/SKILL.md`

**内容**：新增「完成物必须回传」规则

- 使用 `generate-slides`、`generate-infographic`、`generate`（podcast）等生成文件时：
  - 必须在**最终回复末尾**单独一行输出：`MEDIA:<实际输出路径>`
  - 路径以 `--output` 为准；若未指定，用默认路径（如 `./slides.pdf`）
- 若 NotebookLM 返回了 `url`（云端链接），也应输出 `MEDIA:<url>` 或至少把链接写在回复中供主 agent 使用

### 4.2 Subagent 系统 prompt 增强

**位置**：`openclaw/src/agents/subagent-announce.ts` 中的 `buildSubagentSystemPrompt`，或 subagent 的通用 prompt

**内容**：在「Output Format」中明确：

- 若任务生成了文件（PPT、图、音频、视频等），**必须在最终回复末尾**加一行 `MEDIA:<文件路径>`，以便系统自动发回用户
- 若任务产生了可分享链接（如小红书笔记、文档链接），**必须在回复中显式写出该链接**

### 4.3 Subagent announce 流程增强

**位置**：`openclaw/src/agents/subagent-announce.ts` 的 `runSubagentAnnounceFlow`

**内容**：

1. **解析 reply 中的 MEDIA:**  
   对 `reply` 调用 `parseReplyDirectives` 或 `splitMediaFromOutput`，提取 `mediaUrls`
2. **注入 trigger message**：  
   若提取到 mediaUrls，在给主 agent 的 trigger 中：
   - 附加 `MEDIA:<path>` 行（与 findings .md 并列）
   - 或在 instruction 中写明：「Findings 中包含以下生成文件，请在回复中附带：MEDIA:path1, MEDIA:path2」
3. **可选**：若 researcher 通过 exec 生成了文件，可从 transcript 或 tool result 中解析路径，作为补充 media 来源（实现复杂度较高，可作二期）

### 4.4 Internal comm [回报] 支持附件

**位置**：`openclaw/src/agents/subagent-internal-comm.ts`

**内容**：

- `maybeSendInternalCommAnnounce` 增加可选参数 `mediaUrls?: string[]`
- 调用方（`runSubagentAnnounceFlow`）在解析到 mediaUrls 后传入
- `sendToInternalGroup` 已支持 `mediaUrl`/`mediaUrls`，只需透传即可，[回报] 会带附件发到 internal comm group

### 4.5 主 agent 的 announce 指令增强

**位置**：`openclaw/src/agents/subagent-announce.ts` 中的 `triggerMessage` 模板

**内容**：

- 若存在 `mediaUrls`，在 instruction 中明确写：
  - 「用户需要收到以下文件/链接，请在回复末尾按行输出 MEDIA: 路径，不要省略」
  - 列出每个 path，避免主 agent 遗漏

### 4.6 小红书 / 文档等「链接类」完成物

**位置**：xhs-toolkit、相关 skill、以及 subagent prompt

**内容**：

- xhs-toolkit 发布成功后，确保 `final_url` 写入 stdout 或结构化结果，便于解析
- 对应 skill 要求：发布完成后，在回复中**显式写出** `🔗 笔记链接：{final_url}`
- 若为文档类（如 Notion、飞书文档），同理：完成编辑/创建后，必须把文档链接写在回复中

## 五、业界参考

### 5.1 GPT Actions

- 通过 `openaiFileResponse` 返回文件（base64 或临时 URL）
- 文件成为对话的一部分，可被后续使用

### 5.2 LangChain / LangGraph

- `ToolMessage` 支持 `artifact` 参数，可返回 `{ file_path: "/path/to/file.pdf" }`
- 需要下游把 artifact 转为用户可见的附件

### 5.3 Convex Agents

- `storeFile()` 上传文件，`saveMessage()` 保存消息时附带文件引用
- 大文件（>64KB）自动存存储

### 5.4 飞书机器人发文件

- 先调用 `POST /open-apis/im/v1/files` 上传，获得 `file_key`
- 再发消息时引用 `file_key`
- 限制：单文件 30M（图片 10M）

## 六、改动优先级建议

| 优先级 | 改动项 | 影响 | 复杂度 |
|--------|--------|------|--------|
| P0 | nblm SKILL：生成 artifact 后必须输出 MEDIA: | 直接解决 NotebookLM 文件不回传 | 低 |
| P0 | subagent prompt：输出格式要求 MEDIA: | 统一所有 subagent 行为 | 低 |
| P1 | runSubagentAnnounceFlow：解析 reply 的 MEDIA: 并注入 trigger | 主 agent 能收到文件路径 | 中 |
| P1 | trigger message：若有 mediaUrls，明确要求主 agent 附带 | 减少主 agent 遗漏 | 低 |
| P2 | maybeSendInternalCommAnnounce 支持 mediaUrls | [回报] 也带附件到飞书群 | 低 |
| P2 | 小红书/文档 skill：完成必须返回链接 | 链接类完成物可回传 | 低 |

---

## 七、总结

**核心原则**：完成的东西要有回复，不能只有总结。

**实现路径**：
1. **约定**：所有产生文件或链接的 task，执行方必须在回复中显式输出 `MEDIA:<path>` 或链接
2. **解析**：announce 流程解析这些输出，并注入到主 agent 的 trigger 和 internal comm
3. **投递**：沿用现有 MEDIA: → payload.mediaUrls → sendMedia 链路，飞书等 channel 已支持

**不依赖**：无需新增「完成回调」或「artifact 专用通道」，现有 MEDIA: 机制足够，缺的是**约定**和**解析注入**。
