# 上下对齐与任务监控方案

> 让派活、接活、进度、回报像人类协作一样清晰可见，任务不消失、不静默。

---

## 一、问题是什么

你让 Fast「派给 main 做小红书任务」，可能出现：

- Fast 只说「交给 main」，但**没真正派**（没调用 sessions_spawn）
- 派了，但**不知道 main 有没有接到**
- main 在跑，但**跑很久没动静**，不知道是卡住了还是消失了
- 跑完了，有回报，但**中途进度看不到**

就像人类世界：你布置任务，得有人接、有人确认、有人汇报进度，否则任务容易「掉进黑洞」。

---

## 二、方案总览（四层确认）

```
你 → Fast → [派活] → main → [接活] → 执行中 → [进度]（可选）→ [回报]
              ↑         ↑              ↑                ↑
            已有       新增           新增             已有
```

| 层级 | 名称 | 作用 | 状态 |
|------|------|------|------|
| 1 | 派活 | Fast 派任务时，在小弟群发一条「派活了」 | 已有 |
| 2 | 接活 | main 真正开始跑时，发一条「接到了」 | 新增 |
| 3 | 进度 | 跑太久没回报时，定时发「还在跑，已 X 分钟」 | 新增 |
| 4 | 回报 | main 跑完时，发结果摘要 | 已有 |

**多维表格**：不改，保持现状。

**提示词**：不改。这些都是系统自动做的，不靠改 prompt。

**向后兼容**：新功能都是可选的，不配置就不生效，对现有行为无影响。

---

## 三、各层说明（带案例）

### 3.1 派活（已有）

**什么时候**：Fast 调用 sessions_spawn 派任务给 main 时。

**小弟群会看到**：

```
[派活] @main

任务：手动触发小红书任务
Session: agent:main:subagent:xxx-xxx-xxx
```

**案例**：你说「手动触发小红书」，Fast 派给 main，群里立刻出现这条 [派活]。

---

### 3.2 接活（新增）

**什么时候**：main 真正开始执行任务时（系统检测到 run 已 start）。

**小弟群会看到**：

```
[接活] @main

任务：手动触发小红书任务
已开始执行，Session: agent:main:subagent:xxx-xxx-xxx
```

**案例**：派活后几秒，main 开始跑，群里出现 [接活]。这样你就知道：**派出去了，也接到了**。如果只有 [派活] 没有 [接活]，可能是没派出去，或对方没接到。

---

### 3.3 进度（新增）

**什么时候**：任务跑了超过 N 分钟（如 5 分钟）还没回报，系统定时检查，发现还在跑。

**小弟群会看到**：

```
[进度] @main

任务：手动触发小红书任务
进行中，已运行 8 分钟
```

**案例**：小红书任务跑得久，你担心是不是卡住了。系统每 5 分钟查一次，发现还在跑，就发一条 [进度]。你知道「没消失，还在干」。如果超时了，会发 [超时] 并做清理。

**配置**：可配置「超过几分钟才发进度」「多久查一次」，默认可关闭。建议：超过 5 分钟发第一条进度，每 5 分钟检查一次。

**超时**：若任务超过设定时间（如 30 分钟）仍未结束，发 [超时] 并做清理（标记结束、不再等待）。

---

### 3.4 回报（已有）

**什么时候**：main 跑完任务时。

**小弟群会看到**：

```
[回报] @main

任务：手动触发小红书任务
状态：completed successfully

Findings:
（main 的输出摘要）

Stats: runtime 9m • tokens 12k • ...
```

**案例**：main 做完小红书任务，自动发 [回报]，你一眼看到结果和耗时。

**失败时**：若 main 中途出错或超时，[回报] 会发 `状态：failed` 或 `状态：timed out`，并附带错误信息，便于排查。

---

## 四、小弟群消息的指向性（用飞书原生回复）

**目的**：让小弟群里的信息更有指向性——每条 [接活]、[进度]、[回报] 都明确是「针对哪个任务」的回复。

**做法**：利用**飞书已有的回复功能**。发 [接活]、[进度]、[回报] 时，机器人带上「回复哪条消息」的引用，让这些消息以飞书原生的「回复」形式挂在 [派活] 下面——就像你在飞书里点某条消息、选「回复」、再发新消息一样，新消息会显示在被回复消息下方，并带有引用条。

**飞书里的效果**（类似你截图里的样子）：

```
oscar机器人: [派活] @main 任务：手动触发小红书
    │ 回复 oscar机器人: [派活] @main 任务：手动触发小红书
    └─ oscar机器人: [接活] @main 已开始执行

    │ 回复 oscar机器人: [派活] @main 任务：手动触发小红书
    └─ oscar机器人: [进度] @main 进行中，已运行 5 分钟

    │ 回复 oscar机器人: [派活] @main 任务：手动触发小红书
    └─ oscar机器人: [回报] @main 状态：completed successfully
```

**效果**：一眼看出「接活、进度、回报」都是针对「手动触发小红书」这条派活的回复，不会和别的任务混在一起。多个任务同时跑时，每个任务的更新都挂在各自的 [派活] 下面。

**实现要点**：机器人发消息到飞书时，调用飞书的「回复某条消息」接口（如传 root_id / parent_id 等参数），而不是发独立新消息。机器人大概已有这个能力，但目前应未使用，需要接入 internalComm 的发送逻辑。

**前置条件**：回复线程的前提是 [派活] 成功发送到小弟群，且发送接口能返回该消息的 ID（供后续回复引用）。若拿不到 ID，或渠道不支持回复（如微信群），则回退为独立消息，功能仍可用，只是没有线程结构。

## 五、完整流程案例

**场景**：你让 Fast 手动触发小红书任务。

| 时间 | 小弟群 | 说明 |
|------|--------|------|
| 0:00 | [派活] @main 任务：手动触发小红书 | Fast 派了 |
| 0:05 | [接活] @main 任务：手动触发小红书 已开始执行 | main 接到了，开始跑 |
| 5:00 | [进度] @main 进行中，已运行 5 分钟 | 系统检查，发现还在跑 |
| 9:00 | [回报] @main 状态：completed successfully Findings: ... | main 跑完，回报结果 |

这样，派活 → 接活 → 进度 → 回报，全程可见，任务不会「掉进黑洞」。

## 六、派活检测（可选兜底）

**问题**：Fast 有时只说「交给 main」但不调用 sessions_spawn。

**做法**：系统检测 Fast 的回复，若出现「交给 main」「派给 main」等字样，且本轮没有 sessions_spawn 调用，则自动补一次派活，或提示你「未实际派活，是否重试？」。

**优先级**：可放在接活、进度之后再做。先有接活和进度，已经能解决大部分问题。

## 七、实施顺序建议

| 顺序 | 内容 | 说明 |
|------|------|------|
| 1 | 接活确认 | 实现 [接活]，派活后能确认对方接到了 |
| 2 | 进度检查 | 实现 [进度]/[超时]，长时间任务不消失 |
| 3 | 回复线程 | [接活]/[进度]/[回报] 作为对 [派活] 的回复发送，信息有指向性 |
| 4 | 派活检测 | 可选，解决「说了派但没派」 |

**依赖说明**：回复线程（顺序 3）依赖飞书 send 接口能返回 message_id，且支持「回复某条消息」。实施前需确认渠道能力。

**实施状态**：接活确认、进度检查、配置项已实现。回复线程待飞书渠道支持后实现。

---

## 八、配置项汇总（实施时参考）

| 配置项 | 说明 | 建议值 |
|------|------|--------|
| internalCommGroup | 小弟群渠道、群 ID（已有） | 按现有配置 |
| progressCheckIntervalMinutes | 进度检查间隔（分钟），0 表示关闭 | 5（默认启用） |
| progressCheckThresholdMinutes | 超过此分钟数才发 [进度] | 5 |
| runTimeoutMinutes | 超时时间（分钟），超时发 [超时] | 30 |

## 九、边界与注意事项

| 场景 | 说明 |
|------|------|
| **Cron 直接跑 main** | 若 cron 不经过 Fast、直接触发 main，可能没有 [派活]。此时接活、进度、回报可照发，但无法形成回复线程（没有父消息）。可选：cron 触发时也发一条 [派活] 到小弟群。 |
| **其他渠道（非飞书）** | 若 internalCommGroup 是微信群、Telegram 等，可能不支持「回复某条消息」。此时 [接活]/[进度]/[回报] 发独立消息，功能正常，只是没有线程结构。 |
| **多 Agent** | researcher 同理，[派活]/[接活]/[进度]/[回报] 都带 @researcher，逻辑一致。 |
