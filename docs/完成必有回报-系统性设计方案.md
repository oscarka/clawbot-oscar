# 完成必有回报：系统性设计方案

> 一句话：**你交代的事做完了，就要有回音**。不能「做完了就完了」，用户得收到实实在在的东西。

---

## 一、问题是什么？用例子说清楚

### 案例 1：NotebookLM 生成 PPT

- **用户**：在飞书群说「用 NotebookLM 根据这份文档生成个 PPT」
- **实际发生**：研究员跑完任务，PPT 的 PDF 已经生成并保存在本地
- **用户收到的**：只有一句「任务已完成」或「已生成 PPT 大纲」的总结
- **问题**：PDF 文件没发到群里，用户拿不到

### 案例 2：小红书发笔记

- **用户**：让 agent 帮忙发一篇小红书笔记
- **实际发生**：发布成功，笔记有了链接
- **用户收到的**：可能只有「已发布」三个字
- **问题**：笔记链接没给，用户没法点进去看

### 案例 3：文档整理完成

- **用户**：让 agent 整理一份飞书文档
- **实际发生**：文档编辑完成，有可分享链接
- **用户收到的**：只有「整理好了」
- **问题**：文档链接没给，用户不知道去哪看

**共同点**：任务在技术上「完成了」，但**完成物**（文件、链接）没有回到用户手里，等于「沉了」。

---

## 二、OpenClaw 现在是怎么做的？

### 2.1 有流程，但没有「必须回报」的承诺

OpenClaw（[GitHub 仓库](https://github.com/openclaw/openclaw)）已经有「子任务完成 → 发回报」的流程：

- 子 agent（比如研究员）跑完后，会触发一个 **announce** 步骤
- 这个步骤会把结果发回给发起任务的聊天（比如飞书群）

但官方文档写得很清楚：

> 「Announce 的交付是 **best-effort**（尽力而为）；status 为 ok 并**不保证** announce 一定送达。」

也就是说：**有回报机制，但不保证一定回报**。

### 2.2 完成物没有「正式身份」

- 系统没有「完成物」这个正式概念
- 发文件要靠 agent 在回复里写一种特殊格式（类似「MEDIA:文件路径」）
- 如果 agent 没写，系统就不会自动把文件发出去
- 所以：**完成物藏在对话里，系统不会主动去找、去发**

### 2.3 上游最近在做什么（2026 年 3 月）

从 [OpenClaw 最新提交](https://github.com/openclaw/openclaw/commits/main)可以看到：

- **Cron 交付契约**：定时任务现在有更严格的「交付契约」，不再允许随意用主会话兜底，交付方式要写清楚
- **附件与跟进**：ACP 相关改动在加强「附件传递」和「跟进可靠性」
- **空回复处理**：Cron 会区分「空回复」和「不需要回复」，避免误判

说明上游也在往「交付要明确、要可控」的方向走，但**子 agent 的完成物回报**这块，目前还没有类似设计。

## 三、别人家是怎么做的？

### 3.1 A2A 协议（Agent2Agent）

- 把「完成物」当成正式类型，叫 **Artifact**
- 任务完成时，结果里会有一个「产出列表」，每个产出有类型（文本、文件、链接等）
- 不是「顺便带一句」，而是**结构化**的产出

**启示**：完成物要有「名分」，不能只藏在对话里。

### 3.2 SW4RM 协议

- 消息有明确的「处理阶段」：收到 → 已读 → **已完成**
- 「已完成」时可以带上产出位置（比如文件地址）
- 强调「至少送达一次」+ 去重，保证结果可追踪

**启示**：完成要有「已交付」的信号，最好能带产出位置。

### 3.3 业界 UX 研究（2024–2026）

有个说法叫 **「Resolution Without Satisfaction」**（解决了但不满意）：

- 技术上说「做完了」，但用户没拿到东西
- 只给总结、不给原始材料
- 过早说「成功」，其实没验证

对应的设计思路：

- **Completion Commitment**：做到「真正完成」才停，而不是「输出看起来可以」
- **Reflection**：做完后自检，比如检查文件是否真的生成了

**启示**：完成要包含「东西真的到用户手里」，而不只是 run 结束。

## 四、系统性方案：完成承诺框架

### 4.1 核心原则（用人话说）

1. **完成必有回报**：只要产生了用户能用的东西（文件、链接等），就必须有面向用户的回复，不能沉掉
2. **完成物要显式**：文件、链接不能只藏在文字里，要有明确格式，让系统能识别、能转发
3. **可扩展**：以后加新 agent、新业务，只要遵守同一套约定，不用大改系统
4. **可检查**：系统能判断「有没有回报」「回报里有没有完成物」

### 4.2 流程（简化版）

```
用户提需求
    ↓
Agent 执行（可能调用子 agent、工具）
    ↓
产生完成物（文件路径、链接等）
    ↓
按约定格式「登记」完成物（例如在回复末尾写清楚）
    ↓
系统解析这些完成物，注入到要发给用户的消息里
    ↓
用户收到：文字 + 文件/链接
```

### 4.3 约定内容（不写技术细节，只写原则）

**对 Agent / Skill 的约定：**

- 生成了**文件**（PDF、图片、音频、视频等）→ 必须在回复里明确写出文件路径（用约定格式）
- 产生了**链接**（发布链接、文档链接等）→ 必须在回复里显式写出完整链接
- **禁止**：只给总结、不给原始完成物（例如用户要 PPT，不能只给大纲）

**对系统的约定：**

- 回报流程要**解析** agent 回复里的完成物（路径、链接）
- 解析到的完成物要**注入**到发给用户的消息里（不能丢）
- 如果有完成物，内部通知（如飞书群的 [回报]）也要带上，不能只发文字

### 4.4 以后加新 agent 怎么接入？

- 在新 agent 的说明文档里，写清楚：这个 agent 会产生哪些类型的完成物，以及输出格式
- 系统沿用同一套「解析 + 注入」逻辑，不用为每个 agent 单独写一套
- 例如：以后加「设计 agent」，只要在它的说明里写「生成图片后必须输出图片路径」，系统就能自动把图片发回用户

## 五、和 OpenClaw 现状的对应关系

| 我们要的       | OpenClaw 现在       | 需要补什么                         |
|----------------|---------------------|------------------------------------|
| 完成物登记     | 没有                | 从回复里解析路径、链接，形成列表   |
| 回报里带附件   | 只带 findings 文本  | 把解析到的文件、链接注入回报       |
| 统一约定       | 没有                | 在 prompt、SKILL 里写「完成必有回报」 |
| 交付保证       | best-effort         | 先通过解析+注入减少「沉掉」，再考虑重试等 |

## 六、分阶段怎么做（只说思路，不写实现）

### 第一阶段：先立规矩（低成本）

- 在子 agent、主 agent、各 SKILL 的说明里，明确写：**有完成物就必须在回复里写清楚**
- 约定好：文件怎么标、链接怎么标

### 第二阶段：系统会「找」完成物（中成本）

- 回报流程在组消息时，**解析** agent 回复里的路径和链接
- 把解析到的内容**注入**到要发给用户的消息里
- 内部 [回报] 也要支持带附件

### 第三阶段：工具也参与（可选）

- 工具如果生成了文件，在返回结果里标明路径
- 系统把这些路径和 agent 回复里的合并，作为兜底

### 第四阶段：可观测、可保证（可选）

- 统计：有多少任务有完成物、有多少回报里带了完成物
- 若需要更强保证：回报失败时重试、进入死信队列等

## 七、总结

| 问题                         | 答案                                                                 |
|------------------------------|----------------------------------------------------------------------|
| OpenClaw 有没有「完成必有回报」？ | 有回报流程，但**没有**明确契约，交付是 best-effort，完成物没有正式身份 |
| 业界有什么可参考？           | A2A 的 Artifact、SW4RM 的 ACK、UX 的 Completion Commitment          |
| 怎么支持更多 agent？         | 用**统一约定**：各 agent 声明产出格式，系统统一解析、注入             |
| 上游在干嘛？                 | 在加强 Cron 交付契约、附件传递等，子 agent 完成物回报暂无类似设计     |

## 八、结论（直接说清楚）

### 现状结论

1. **问题确实存在**：NotebookLM 的 PPT、小红书的链接、文档的链接，都会「沉掉」—— 任务做完了，但用户收不到文件或链接。
2. **OpenClaw 没兜住**：有回报流程，但完成物要靠 agent 自己写对格式，系统不主动找、不主动发。上游也没专门做这块。
3. **根因**：没有「完成必有回报」的硬性约定，完成物没有正式身份，全靠 agent 自觉。

### 方案结论

**要解决，需要两件事：**

1. **立规矩**：在 prompt、SKILL 里写死——「有文件/链接就必须在回复里写清楚，否则不算完成」。
2. **系统兜底**：回报流程要**解析**回复里的路径和链接，**注入**到发给用户的消息里，不能指望 agent 一定写对。

### 行动结论

**建议先做：**

- **第一阶段**：在 nblm SKILL、subagent prompt、主 agent 的 announce 指令里，加上「完成必有回报」的规则（改文案、改约定，成本低）。
- **第二阶段**：改回报流程，让它解析 reply 里的完成物并注入到消息里（改系统逻辑，成本中）。

**不做会怎样**：继续靠 agent 自觉，PPT、链接还是会沉，用户还是会问「东西呢？」。

## 九、替代方案：低调试成本思路

原方案依赖「agent 在回复里写对格式」+「系统解析 agent 文本」，调试时要排查：是 prompt 没写清楚？模型没遵守？解析正则漏了？层级多、变量多，成本高。

### 核心思路：**不靠 agent，靠工具**

完成物是谁产生的？**工具**。工具（如 artifact_manager、exec）跑完就知道自己生成了什么，输出是确定的，不像 agent 文本那样随机。

**把「完成物从哪来」从 agent 文本改成工具结果**，调试会简单很多。

### 方案 A：约定输出目录（推荐，成本最低）

下面把方案 A 说完整、说严谨，但用人话讲。

---

#### 1. 一句话概括

**工具生成的文件，统一写到一个「约定好的目录」；任务结束时，系统扫这个目录，把里面的东西全部发给用户。** 不靠 agent 写对格式，不解析对话内容。

---

#### 2. 目录长什么样？

约定一个根目录，例如：

`~/.openclaw/artifacts/`

每次任务（子 agent 的一次 run）有一个唯一 ID，下面再分子目录：

`~/.openclaw/artifacts/<任务ID>/`

例如：`~/.openclaw/artifacts/abc123-def456/`

- 任务 ID 用这次 run 的 ID 即可，系统在跑任务时就知道
- 一个任务一个目录，互不串

---

#### 3. 工具要怎么做？

**会生成文件的工具**（如 NotebookLM 的 artifact_manager、生成图片的脚本等）：

- 跑之前，检查有没有「约定目录」的环境变量（例如 `OPENCLAW_ARTIFACTS_DIR`）
- 若有，就把生成的文件写到这个目录里（而不是写到别的地方）
- 若没有，按原来的逻辑（如 `--output` 指定路径）照常写

**会生成链接的工具**（如小红书发布）：

- 链接不是文件，但可以写到一个「清单文件」里
- 例如在约定目录下放一个 `links.json`，内容类似：`[{"label": "小红书笔记", "url": "https://..."}]`
- 回报流程扫目录时，除了发文件，也读这个清单，把链接附在消息里

**谁负责把「约定目录」传给工具？**

- 执行工具的那一层（例如 exec 或 mcporter 的包装）在跑命令前，先把「这次任务的目录」设好，通过环境变量传给子进程
- 工具只需读环境变量，不需要自己算任务 ID

---

#### 4. 回报流程要怎么做？

子 agent 跑完后，在发回报之前：

1. 找到这次任务对应的目录
2. 看目录里有没有文件、有没有 `links.json`
3. 若有，把文件全部附在要发给用户的消息里，把链接也写上
4. 发完（或发失败）后，可以删掉这个目录，或留一段时间再清理，避免占磁盘

---

#### 5. 边界情况怎么处理？

| 情况 | 处理方式 |
|------|----------|
| 目录里没有文件 | 正常发文字回报，不报错 |
| 目录里有多个文件 | 全部附带，按顺序发 |
| 文件太大（超过飞书等限制） | 可跳过或只发链接，不硬塞；日志记一笔 |
| 目录不存在 | 当作「这次没有完成物」，正常发文字回报 |
| 链接清单格式不对 | 忽略，只发文件；不因一个文件坏掉整次回报 |
| 主 agent 直接跑（不是子 agent） | 若有任务 ID，逻辑一样；若无，可沿用现有行为 |

---

#### 6. 新增工具怎么接入？

以后加新工具（如「设计 agent」生成图片）：

- 让工具在生成文件时，检查 `OPENCLAW_ARTIFACTS_DIR`
- 有就写到该目录，没有就写原来地方
- 不碰 prompt、不碰 agent，不解析文本

---

#### 7. 调试为什么简单？

- **文件没发出去？** 先看约定目录里有没有文件。有 → 问题在「回报流程」；没有 → 问题在「工具没写进去」。
- **链接没发出去？** 看 `links.json` 有没有、内容对不对。
- 不用猜：是 prompt 问题？模型问题？解析问题？层级一清二楚。

---

#### 8. 要改哪些东西？（实现成本）

| 改动点 | 内容 |
|--------|------|
| 执行层（exec wrapper 等） | 跑命令前，创建约定目录并写入环境变量 |
| 各产出工具 | 读环境变量，有则写约定目录 |
| 回报流程 | 扫约定目录，把文件和链接注入消息 |
| 清理逻辑（可选） | 发完后删目录或定期清理 |

不改 prompt、不改 agent 行为、不解析 agent 文本。

---

#### 9. 小结（方案 A 的要点）

- **约定**：工具往「约定目录」写，回报流程从约定目录读。
- **责任清晰**：工具负责写对地方，系统负责扫目录、发出去。
- **调试简单**：出问题先看目录，有或没有，立刻能判断是哪一层。
- **可扩展**：新工具只需遵守「有环境变量就写约定目录」，不用改别处。

### 方案 B：工具结果里带「产出路径」

**做法**：工具返回时增加约定字段，例如 `outputFiles: ["/path/to/slides.pdf"]`。

- exec 包装：若 stdout 里有「Saved to: xxx」，解析出来填进 `outputFiles`
- 回报流程：从 transcript 里读最近几次工具调用的结果，收集所有 `outputFiles`，注入消息

**优点**：
- 不依赖 agent 文本
- 工具输出是结构化的，解析稳定
- 调试：看工具 result 里有没有 outputFiles

**成本**：改 exec 或各工具的返回格式；回报流程要读 transcript 并解析 tool result。

### 方案 C：agent 文本 + 工具结果双轨（折中）

- **主通道**：从工具结果 / 约定目录取完成物（方案 A 或 B）
- **兜底**：仍解析 agent 回复里的 MEDIA:，作为补充

工具没遵守约定时，至少 agent 写对了还能发出去。

---

### 对比：原方案 vs 低调试成本方案

| 维度       | 原方案（靠 agent 文本）     | 低调试成本方案（靠工具）       |
|------------|----------------------------|--------------------------------|
| 完成物来源 | agent 在回复里写 MEDIA:    | 工具写入约定目录 / 返回字段    |
| 是否依赖 LLM | 是，要模型「记得」写对格式 | 否，工具输出是确定的           |
| 调试方式   | 查 prompt、模型、解析、多层 | 查目录有没有文件 / 工具 result |
| 新增 skill | 每个要写规则、测 prompt    | 工具遵守约定即可               |
| 实现成本   | 改 prompt + 改解析逻辑     | 改工具写入逻辑 + 改回报扫描    |

**结论**：若希望调试成本低，优先考虑**方案 A（约定输出目录）**。上面已把方案 A 写完整：目录结构、工具怎么接入、回报怎么扫、边界怎么处理、调试怎么查。核心就一句：**工具往约定目录写，系统扫目录发，不靠 agent，不解析文本。**
