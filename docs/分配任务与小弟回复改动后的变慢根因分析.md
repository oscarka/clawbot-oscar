# 分配任务 + 小弟回复改动后的变慢根因分析

> 用户反馈：自从引入「分配任务 + 小弟回复」之后，**所有渠道**（Feishu、电脑端等）都变慢了，不只是 clawhub。

---

## 一、改动定位

相关改动集中在 **359eaeb**（checkpoint: before subagent internal comm group feature）及之前的提交。

### 1.1 核心改动：system-prompt.ts（+110 行）

```diff
# 359eaeb 对 system-prompt.ts 的改动
- 原有较短的 小红书 说明（~15 行）
+ 按 toolNames 分支：
+   - 有 exec：~80 行 NotebookLM / XHS / Android / Feishu 文件处理
+   - 无 exec：~20 行 sessions_spawn 强制调用说明（FAST agent）
```

**关键逻辑**（`system-prompt.ts` 约 409 行）：

```ts
...(params.toolNames?.includes("exec") ? [
  // 长块：NotebookLM 全功能、小红书、Android、Feishu 文件处理
  // 约 80 行
] : [
  // 短块：sessions_spawn 强制调用（FAST agent）
  // 约 20 行
]),
```

- **有 exec 的 agent**（main / 全功能）：得到 ~80 行密集指令
- **无 exec 的 agent**（fast）：得到 ~20 行 sessions_spawn 指令

### 1.2 其他相关改动

- `sessions-spawn-tool.ts`：派活后 `maybeSendInternalCommSpawn` 等
- `subagent-internal-comm.ts`：派活 / 接活 / 进度 / 回报 到小弟群
- `subagent-registry.ts`：接活、进度检查、超时、回报流程

## 二、变慢根因（按影响排序）

### 2.1 【主因】System Prompt 膨胀 → 首 token 延迟增加

| 路径 | 新增行数 | 估算新增 token | 影响 |
|------|----------|----------------|------|
| 有 exec（main） | ~80 行 | ~1500–2500 | 所有使用 main agent 的渠道 |
| 无 exec（fast） | ~20 行 | ~300–500 | 使用 fast agent 的渠道 |

**机制**：

1. 每次请求都会带上完整 system prompt
2. 输入 token 越多，模型处理越慢，首 token 越晚
3. 新增内容多为长指令（NotebookLM、小红书、Android 等），token 密度高

**结论**：这是「所有渠道都变慢」的主要来源，与是否实际调用 sessions_spawn 无关。

---

### 2.2 【次因】队列默认 collect + debounce 1s

- 默认 `QueueMode`：`collect`
- 默认 `DEFAULT_QUEUE_DEBOUNCE_MS`：1000ms

在 collect 模式下，首条消息会先进入队列，再等 debounce 再 drain。  
对单条消息而言，会多出最多约 1 秒的等待，再开始跑 agent。

---

### 2.3 【非主因】sessions_spawn / 小弟回复 本身不阻塞主流程

- `sessions_spawn`：`callGateway(agent)` 返回 `runId` 即结束，不等待 subagent 完成
- `waitForSubagentCompletion`：`void` 调用，不阻塞
- `maybeSendInternalCommSpawn` 等：`void sendToInternalGroup`，不阻塞
- `runSubagentAnnounceFlow`：在 subagent 结束后异步执行，不阻塞主 agent 的首次回复

因此，变慢不是由「等待小弟回复」或「等待派活完成」造成的。

---

### 2.4 【非主因】internalCommGroup 发送

- 派活 / 接活 / 进度 / 回报 都是 `void` 异步发送
- 不阻塞主 agent 的请求与回复路径

## 三、数据流简要梳理

```
用户发消息
    ↓
dispatchReplyFromConfig → getReplyFromConfig
    ↓
runPreparedReply → runReplyAgent
    ↓
buildEmbeddedSystemPrompt(tools) → buildAgentSystemPrompt(toolNames)
    ↓
toolNames.includes("exec") ?
    → 有 exec：~80 行长块（NotebookLM/XHS/Android 等）
    → 无 exec：~20 行 sessions_spawn 块
    ↓
Pi 发起 agent 请求（system prompt 已膨胀）
    ↓
LLM 处理更多输入 token → 首 token 延迟增加
```

## 四、优化建议

### 4.1 优先：缩短 system prompt

1. **按需加载**：NotebookLM / XHS / Android 等指令按场景注入，而不是每次全量
2. **条件触发**：仅在用户消息涉及相关关键词时，才加入对应指令
3. **精简表述**：合并重复说明，缩短单条指令长度

### 4.2 次优：调整队列与 debounce

1. 对单条消息：`mode: "queue"` 或 `mode: "immediate"` 减少 debounce 延迟
2. 按渠道调整 `debounceMs`：对需要快速响应的渠道可设为 0 或 300–500ms

### 4.3 可选：拆分 agent 与 prompt

- 将「研究型任务」交给 researcher agent；main agent 保持轻量 prompt
- 减少 main agent 的 tool 列表和指令，从而降低 prompt 体积

---

## 五、小结

| 项目 | 说明 |
|------|------|
| **主因** | system-prompt 增加 ~80–110 行，导致每次请求多 1500–2500 token，首 token 延迟上升 |
| **次因** | 默认 collect + 1s debounce，增加首条消息的等待时间 |
| **非主因** | sessions_spawn、小弟回复、internalComm 均为异步，不阻塞主流程 |
| **影响范围** | 所有使用 main agent 的渠道；使用 fast agent 的渠道也有一定影响 |

**结论**：变慢主要来自「分配任务 + 小弟回复」改动引入的 **system prompt 膨胀**，而不是派活、接活、回报流程本身。优化方向应集中在 prompt 精简和按需加载。
