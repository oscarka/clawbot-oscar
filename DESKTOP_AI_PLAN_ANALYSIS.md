# 桌面 AI 实现方案：理解、差距分析与落地丰富文档

> 本文档面向非技术人员与 AI 实现方（如 Cursor、Antigravity），对「桌面 AI 完整实现方案」进行理解、差距分析，并补充落地细节。**不含代码，仅分析与说明。**

---

## 一、方案理解与核心要点

### 1.1 总体目标理解

| 目标 | 含义 | 实现难点 |
|------|------|----------|
| **用户友好** | 用户只需启动 AI、授权权限，极少理解原理 | 需要清晰的首次引导、权限弹窗、异常提示 |
| **极少人工** | 技术团队只在部署、异常、大更新时介入 | 需要自动化监控、自愈、异常分级 |
| **高自动化** | AI 自动操作，无需人工干预 | 需要 AX 主力 + 视觉回退 + 动态适配 |
| **可维护** | 换电脑、App 更新、新 App 接入可自动适配 | 需要 AX 树缓存、语义映射、自动刷新 |
| **稳定性高** | UI 微调不导致操作失败 | 需要语义查询 + 多路径回退 |

### 1.2 架构理解

```
用户 → 启动 AI / 授权 → 桌面 AI App / MCP 接口
                          │
                          ├─ AX 语义树 + 缓存（主力）
                          │   ├─ 异步多窗口
                          │   └─ 动态 UI 监听
                          │
                          └─ 视觉回退（仅在 AX 失败时）
                             └─ YOLO / 截图 + AI 识别
```

- **主力路径**：AX 语义树 + 缓存，语义查询驱动操作，速度快、稳定。
- **回退路径**：AX 失败时再用截图 + AI 识别，保证成功率。
- **接口形态**：用户通过 MCP 或桌面 App 发起「语义动作」（如「点击发送」），无需关心底层实现。

### 1.3 关键设计理念

1. **AX 优先，视觉为辅**：控件用 AX 精确定位，视觉只在 AX 不可用时兜底。
2. **语义化操作**：上层只表达「点击发送」「输入收件人」等语义，不关心坐标或 AXID。
3. **缓存加速**：常用控件 AXID 缓存，减少全树遍历。
4. **动态适配**：局部刷新 AX 树、检测 UI 变化，自动更新缓存。
5. **异步与多窗口**：支持多应用、多窗口并行操作，提升体验。

## 二、与当前实现的差距分析

### 2.1 当前实现概览

基于对 moltbot 项目的分析，当前实现大致如下：

| 模块 | 当前实现 | 方案要求 | 差距 |
|------|----------|----------|------|
| **操作主力** | 视觉优先（截图 + AI 识别） | AX 优先，视觉回退 | **架构相反**：当前以视觉为主，无 AX 主力路径 |
| **AX 语义树** | 仅有 `inspect_ui_poc.py` 演示飞书 AX 遍历 | 完整 AX 树 + 语义节点图 + 缓存 | **未集成**：AX 仅作 POC，未进入主流程 |
| **语义查询** | 无 | `semantic_query("button", title="发送")` | **缺失**：无语义查询层 |
| **控件缓存** | 无 | 常用控件 AXID/semantic ID 缓存 | **缺失** |
| **视觉回退** | 视觉是主路径 | 仅 AX 失败时触发 | **角色颠倒**：视觉不是回退，而是主方案 |
| **MCP 接口** | 无桌面自动化 MCP | 语义动作 → 操作结果 | **缺失** |
| **动态 UI** | 每次全屏截图 | 局部刷新 AX 树、快照对比 | **缺失** |
| **异步多窗口** | 顺序执行 | 并行多窗口/多应用 | **缺失** |
| **用户提示** | 有 | 权限、异常、完成通知 | **部分具备**，需与方案对齐 |

### 2.2 具体差距说明

#### 2.2.1 操作路径：主辅颠倒

- **方案**：AX 为主，视觉为辅。
- **当前**：`vision_agent.py` 以截图 + AI 视觉分析为主，通过坐标（cliclick/peekaboo）执行操作。
- **影响**：视觉路径延迟大、易受 UI 变化影响，稳定性不如 AX。

#### 2.2.2 AX 使用：仅 POC，未集成

- **方案**：抓取 AX 树、解析窗口/控件、生成语义节点图、缓存常用控件。
- **当前**：`inspect_ui_poc.py` 能遍历飞书 AX 树（role、title、frame），但未接入主流程。
- **影响**：未利用 AX 的稳定性和语义信息，也未建立缓存。

#### 2.2.3 语义层：无抽象

- **方案**：上层发出「click 发送」「input 收件人」等语义指令，底层通过语义查询找到控件并执行。
- **当前**：任务规划输出自然语言目标（如「顶部工具栏左侧的加号按钮」），视觉模型直接输出坐标，无语义查询层。
- **影响**：没有稳定的「控件语义 → AX 操作」映射，难以做缓存和复用。

#### 2.2.4 回退机制：无 AX→视觉 回退

- **方案**：AX 失败时自动切换到视觉（截图 + YOLO/识别），并尝试更新 AX 图。
- **当前**：`test_vision_recovery.py` 有「主操作失败 → 分析反馈 → 执行恢复操作」的逻辑，但只基于视觉，没有 AX 失败检测。
- **影响**：缺少「AX 失败 → 视觉兜底」的完整回退链。

#### 2.2.5 接口形态：无 MCP

- **方案**：通过 MCP 暴露「语义动作 → 操作结果」接口，供 LLM/Agent 调用。
- **当前**：通过 Python 脚本直接调用 `VisionAgent`，无 MCP 或标准化接口。
- **影响**：不易与 Cursor、Antigravity、OpenClaw 等 Agent 体系集成。

#### 2.2.6 已有能力（可复用）

- **Peekaboo**：截图、点击、输入、窗口管理、菜单等，已集成。
- **视觉分析**：GLM-4.6v-flash 等分析截图、定位控件，可作为回退路径。
- **任务规划**：Claude 等规划任务步骤，可继续使用。
- **应用查找**：`find_app_intelligently` 等逻辑可保留。
- **错误恢复**：`analyze_realtime_feedback` 等可并入 AX→视觉 回退流程。

## 三、落地内容丰富与细化

### 3.1 AX 语义树与缓存

#### 3.1.1 语义树抓取

- **触发时机**：首次接入新 App、App 大版本更新、用户手动「刷新控件」。
- **输出**：结构化 JSON，包含应用名、窗口、控件类型、标题、位置、层级等。
- **示例结构**（与方案一致）：
  ```json
  {
    "window": "Mail",
    "children": [
      {"type": "button", "title": "发送", "AXID": "btn123"},
      {"type": "textField", "title": "收件人", "AXID": "tf456"}
    ]
  }
  ```
- **人工参与**：技术团队首次接入时确认关键控件 AXID 是否正确。

#### 3.1.2 语义节点图

- **作用**：将 AX 树转为可查询的语义图（按类型、标题、父级等）。
- **查询形式**：如 `type=button AND title=发送`、`type=textField AND title=收件人`。
- **层级信息**：保留父子关系，便于处理 Tab、弹窗、滚动区域等。

#### 3.1.3 缓存设计

- **缓存内容**：常用操作对应的 AXID / semantic ID。
- **更新策略**：局部刷新（只刷新当前窗口/当前 Tab），定期或按需全量刷新。
- **失效条件**：App 重启、窗口关闭、大版本更新、操作失败。
- **存储**：本地 JSON/数据库，按 App + 窗口 + 版本 组织。

### 3.2 语义查询与 AX 操作

#### 3.2.1 语义查询流程

1. 接收语义动作（如「点击发送」）。
2. 先查缓存：是否存在对应 AXID。
3. 缓存命中 → 直接执行 AX 操作。
4. 缓存未命中 → 在 AX 语义树中查询（type、title、父级等）。
5. 查询成功 → 执行并更新缓存。
6. 查询失败 → 触发视觉回退。

#### 3.2.2 AX 操作类型

- **AXPress**：按钮、链接等可点击控件。
- **AXSetValue**：输入框、下拉框等。
- **AXIncrement/AXDecrement**：滑块、步进器等。
- **其他**：根据 macOS AX API 支持的 action 扩展。

#### 3.2.3 容错策略

- 操作超时：重试 1–2 次，仍失败则回退视觉。
- 控件不可见：尝试滚动、切换 Tab、关闭弹窗后再查。
- 多匹配：按可见性、位置、最近使用等排序，选最可能的一个。

### 3.3 视觉回退机制

#### 3.3.1 触发条件

- AX 语义查询无结果。
- AX 操作执行失败（超时、错误码）。
- 验证步骤发现界面与预期不符（可选）。

#### 3.3.2 回退流程

1. 截图当前界面。
2. 使用 YOLO/图像识别或 AI 视觉模型定位目标控件。
3. 获取坐标，执行点击/输入。
4. 若成功，尝试更新 AX 语义图（将新发现的控件加入）。
5. 更新缓存（如可用）。

#### 3.3.3 用户参与

- 无需用户参与，全部自动执行。
- 若视觉也失败，可弹窗提示用户或上报技术团队。

### 3.4 动态 UI 处理

#### 3.4.1 场景

- 滚动视图：列表、聊天记录等。
- Tab 页：多 Tab 窗口。
- 弹窗：对话框、下拉菜单、toast。

#### 3.4.2 策略

- **局部刷新**：只刷新当前窗口/当前 Tab 的 AX 子树，不全树遍历。
- **快照对比**：对比前后 AX 树或关键节点，检测 UI 变化。
- **增量更新**：仅更新变化部分，保留未变控件的缓存。
- **用户参与**：零，全自动。

### 3.5 异步与多窗口操作

#### 3.5.1 场景

- 同时在多个应用间操作（如从飞书复制到邮件）。
- 同一应用多窗口并行处理。
- 后台执行不影响用户当前操作。

#### 3.5.2 实现要点

- 使用 asyncio / 多线程 / Swift 异步。
- 按窗口/应用分配任务，避免互相阻塞。
- 保证操作顺序依赖（如 A 完成后再执行 B）。
- 用户参与：零。

### 3.6 MCP / LLM 接口

#### 3.6.1 输入

- 语义动作：`click`、`input`、`select`、`scroll` 等。
- 目标描述：如「发送按钮」「收件人输入框」。
- 可选：应用名、窗口标题、上下文。

#### 3.6.2 输出

- 操作结果：成功/失败。
- 状态反馈：当前界面、错误信息。
- 可选：截图、AX 树片段（调试用）。

#### 3.6.3 设计原则

- AI 只表达「想做什么」，不关心底层控件或坐标。
- 接口与 Cursor、Antigravity、OpenClaw 等 Agent 体系兼容。

### 3.7 用户友好提示

#### 3.7.1 权限提示

- 首次运行：明确说明需要辅助功能、屏幕录制权限。
- 提供跳转到系统设置的链接或按钮。
- 检测权限状态，未授权时给出操作指引。

#### 3.7.2 操作指示

- 异常：弹窗说明失败原因及建议（如「未找到发送按钮，请确认当前在聊天界面」）。
- 需确认：弹窗让用户确认是否继续（如敏感操作）。
- 完成：任务完成后清晰通知（通知中心、状态栏、弹窗等）。

#### 3.7.3 首次使用引导

- 简要说明「一次授权，后续自动运行」。
- 可选：简单演示任务（如「打开飞书并发送一条消息」）。

### 3.8 智能自适应机制

#### 3.8.1 换电脑 / 新账号

- 自动抓取 AX 树并生成缓存。
- 无需用户操作，技术团队仅在必要时介入。

#### 3.8.2 App 小更新

- 自动检测 AX 变化（如标题、结构微调）。
- 自动校正节点、更新缓存。
- 用户无感知。

#### 3.8.3 App 大更新

- 检测到结构性变化（如大面积 AX 树变化）时。
- 技术团队少量确认关键控件。
- 用户一般无需操作。

#### 3.8.4 异常处理

- 自动尝试视觉回退。
- 失败时通知技术团队，并记录日志、截图等。
- 可选：提供「一键上报」给用户。

## 四、落地实现顺序（细化）

### 阶段 1：部署与环境准备

1. 授权辅助功能、屏幕录制。
2. 确认 Python / Swift 运行环境。
3. 部署 MCP 或桌面 App 入口（供后续接入）。

### 阶段 2：首次 AX 树抓取与缓存

1. 实现 AX 树抓取（可参考 `inspect_ui_poc.py`）。
2. 定义语义节点图结构（JSON schema）。
3. 技术团队抓取关键 App 的关键控件 AXID。
4. 实现语义图构建与缓存写入。

### 阶段 3：语义操作与视觉回退

1. 实现语义查询函数（缓存优先 → AX 树查询）。
2. 实现 AX 操作封装（AXPress、AXSetValue 等）。
3. 实现视觉回退：AX 失败 → 截图 → 识别 → 执行 → 更新 AX（可选）。

### 阶段 4：动态 UI 与异步操作

1. 实现局部 AX 树刷新。
2. 实现 UI 变化检测（快照对比）。
3. 实现多窗口/多应用并行执行框架。

### 阶段 5：用户友好与智能自适应

1. 权限弹窗与引导。
2. 异常与完成通知。
3. 换电脑、小更新自动适配。
4. 大更新时少量人工确认流程。

## 五、预期效果与验证指标

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| 用户认知成本 | 极低，几乎零学习 | 新用户完成首次任务所需步骤数 |
| 用户操作量 | 仅启动 + 少量确认 | 单次任务中用户交互次数 |
| 技术团队工作量 | 部署 + 异常确认，日常少干预 | 每周/每月人工干预次数 |
| 操作速度 | AX 为主，视觉为辅 | 单步操作平均耗时 |
| 稳定性 | 大部分 UI 变化可自动适配 | 任务成功率、回退触发比例 |
| 扩展性 | 新 App 可快速接入 | 新 App 接入所需人工步骤 |

## 六、总结

### 6.1 方案理解

方案以 **AX 主力 + 视觉回退** 为核心，通过 **语义查询 + 缓存** 实现高稳定、可维护的桌面自动化，并借助 **MCP 接口** 与上层 AI 体系集成。目标是用户接近零认知、技术团队极少干预、AI 全程自动化执行。

### 6.2 当前差距

当前实现以 **视觉为主**，AX 仅作 POC，未形成语义查询、缓存、MCP 等完整链路。需要做 **架构调整**：建立 AX 主力路径，将视觉定位为回退，并补充语义层、缓存、MCP 和动态 UI 能力。

### 6.3 落地重点

1. 建立 AX 语义树与缓存体系。
2. 实现语义查询 → AX 操作 → 视觉回退的完整链路。
3. 提供 MCP 或标准化接口，便于与 Cursor / Antigravity 集成。
4. 完善用户提示与智能自适应机制。
5. 在现有 Peekaboo、视觉分析、任务规划基础上，逐步迁移至新架构。
