# OpenClaw 智能卡顿检测说明

## 问题：如何区分"卡顿"和"正常处理中"？

这是一个关键问题。如果误判，可能会在正常处理时错误地停止任务。

## 当前脚本的改进

### 改进前的问题

原来的脚本只检查 `agent.wait` 的时间，但这不够准确：
- `agent.wait` 时间长可能是正常处理（比如复杂任务需要时间）
- 没有检查是否有活动迹象
- 可能误判正常处理为卡顿

### 改进后的检测逻辑

现在的脚本使用多重指标来判断：

#### 1. 优先检查 OpenClaw 自己的标记

OpenClaw 系统本身会标记 "stuck session"，这是最可靠的指标：
- 如果检测到 "stuck session" 日志 → 立即停止（这是系统自己判断的）

#### 2. 检查活动迹象

在判断 `agent.wait` 时间过长时，会先检查是否有活动迹象：

**活动迹象包括：**
- 最近5分钟内有工具调用（tool events）
- 最近5分钟内有流式输出（stream events）
- 最近5分钟内有助手回复（assistant deltas）
- 最近5分钟内有生命周期事件（lifecycle events）
- 日志文件最近有更新

**判断逻辑：**
- 如果 `agent.wait` 超过120秒 **且** 没有活动迹象 → 判定为卡顿，自动停止
- 如果 `agent.wait` 超过120秒 **但** 有活动迹象 → 判定为正常处理中，继续观察
- 如果 `agent.wait` 超过60秒但小于120秒 → 警告，但检查活动迹象后再决定

#### 3. 检查连续超时

连续超时通常是真正的错误，会立即处理：
- 检测到3次以上连续超时 → 重置会话

## 检测指标说明

### 卡顿的典型特征

1. **长时间无响应**
   - `agent.wait` 超过120秒
   - 没有任何活动迹象（工具调用、流式输出等）
   - 日志文件长时间无更新

2. **系统标记**
   - OpenClaw 自己标记的 "stuck session"
   - 这通常发生在 processing 状态超过120秒时

3. **连续错误**
   - 连续3次以上超时
   - 大量错误日志

### 正常处理中的特征

1. **有活动迹象**
   - 有工具调用（tool events）
   - 有流式输出（stream events）
   - 有助手回复（assistant deltas）
   - 日志文件持续更新

2. **合理的等待时间**
   - 复杂任务可能需要较长时间
   - 但会有中间输出或活动

3. **生命周期正常**
   - 有 lifecycle 事件
   - 状态在正常流转

## 阈值设置

### 当前阈值

- **警告阈值**：60秒
  - 如果 `agent.wait` 超过60秒，会记录警告
  - 但会检查活动迹象，不会立即停止

- **严重阈值**：120秒
  - 如果 `agent.wait` 超过120秒且无活动迹象，会停止任务
  - 如果有活动迹象，会继续观察

- **连续超时阈值**：3次
  - 连续3次超时会重置会话

### 如何调整阈值

如果需要调整，可以修改脚本中的这些变量：

- `AGENT_WAIT_WARNING=60` - 警告阈值（秒）
- `AGENT_WAIT_CRITICAL=120` - 严重阈值（秒）
- `MAX_CONSECUTIVE_TIMEOUTS=3` - 连续超时阈值（次）

## 实际效果

### 场景1：正常处理复杂任务

- `agent.wait` 可能超过120秒
- 但会有工具调用、流式输出等活动
- **结果**：不会误判，继续观察

### 场景2：真正卡住

- `agent.wait` 超过120秒
- 没有任何活动迹象
- 日志文件长时间无更新
- **结果**：判定为卡顿，自动停止

### 场景3：系统标记的卡住

- OpenClaw 自己标记了 "stuck session"
- **结果**：立即停止，最可靠

## 注意事项

1. **活动检测的时间窗口**
   - 当前设置为5分钟（300秒）
   - 如果任务处理时间超过5分钟，可能需要调整

2. **日志时间戳解析**
   - 脚本会尝试解析日志时间戳
   - 如果时间戳格式不匹配，可能无法准确判断

3. **误判的可能性**
   - 虽然改进了，但仍有可能误判
   - 建议定期检查监控日志，根据实际情况调整阈值

## 监控日志

所有判断都会记录到监控日志中：
- `~/.openclaw/monitor.log`

可以查看日志了解脚本的判断过程：
```bash
tail -f ~/.openclaw/monitor.log
```

日志会显示：
- 检测到的活动迹象
- 判断过程
- 最终决定（停止/继续观察）

## 总结

改进后的脚本能够：
- ✅ 区分"卡顿"和"正常处理中"
- ✅ 优先使用 OpenClaw 自己的标记
- ✅ 检查活动迹象，避免误判
- ✅ 记录详细的判断过程

这样可以在不干扰正常处理的同时，及时处理真正的卡顿问题。
